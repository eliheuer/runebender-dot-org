<DocSection title="Choose the right report">

- Use Core issues for shared editing state, data model, undo, selection, and cross-front-end behavior.
- Use Xilem issues for native editor build failures, crashes, UFO behavior, tracing behavior, tools, shortcuts, and format problems.
- Use Comfy issues for ComfyUI nodes, Vue/WASM editor behavior, workspace-backed `FONT` wires, and graph workflows.
- Use website issues or patches for stale docs, broken links, unclear examples, and agent-readable context files.
- Use Zulip for early design discussion, contributor coordination, and questions that are not yet issue-shaped.
- Use pull requests when you can make a small, reviewable code or docs change.

</DocSection>

<DocSection title="Useful report shape">

For alpha software, a small precise report is better than a broad complaint.

<table>
  <caption>Include</caption>
  <tbody>
    <tr><th>Context</th><td>Source checkout or commit, operating system, Rust version if build-related.</td></tr>
    <tr><th>Command</th><td>The exact <code>cargo run</code> command or app action.</td></tr>
    <tr><th>Data</th><td>Example UFO path, glyph name, image input, or hyperbezier file if relevant.</td></tr>
    <tr><th>Behavior</th><td>Expected result, actual result, and whether undo or reopening changed anything.</td></tr>
    <tr><th>Scope</th><td>Whether it affects local tracing, QuiverAI, normal cubic outlines, hyperbeziers, or docs.</td></tr>
  </tbody>
</table>

</DocSection>
